# 데이터 패칭

## 핵심 원칙

모든 API 호출은 `src/lib/api.ts` 의 axios 싱글톤을 통해 이루어집니다.
Next.js Route Handler나 Server Action은 사용하지 않습니다.

## axios 싱글톤 사용

```tsx
// ✅ 항상 @/lib/api 에서 import
import api from '@/lib/api'

useEffect(() => {
  api.get('/inks')
    .then(res => setInks(res.data))
    .catch(err => console.error(err))
    .finally(() => setLoading(false))
}, [])
```

```tsx
// ❌ 직접 fetch 또는 axios import 금지
import axios from 'axios'
const res = await axios.get('http://localhost:8001/inks')

// ❌ URL 하드코딩 금지
fetch('http://localhost:8001/inks')
```

## 표준 데이터 패칭 패턴

`useEffect` + `useState` 조합으로 로딩/에러 상태를 관리합니다.

```tsx
'use client'
import { useEffect, useState } from 'react'
import api from '@/lib/api'

interface InkItem {
  id: number
  name: string
  category: string
}

export default function MasterPage() {
  const [inks, setInks] = useState<InkItem[]>([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    api.get<InkItem[]>('/inks')
      .then(res => setInks(res.data))
      .catch(err => console.error('잉크 목록 로드 실패:', err))
      .finally(() => setLoading(false))
  }, [])

  if (loading) return <div>로딩 중...</div>
  return <ul>{inks.map(ink => <li key={ink.id}>{ink.name}</li>)}</ul>
}
```

## 병렬 요청

여러 독립적인 API를 동시에 호출할 때는 `Promise.all` 을 사용합니다.
(보색 도메인의 3-method 비교 패턴 참고)

```tsx
// ✅ 병렬 요청
useEffect(() => {
  setLoading(true)
  Promise.all([
    api.post('/complementary/method1', payload),
    api.post('/complementary/method2', payload),
    api.post('/complementary/method3', payload),
  ])
    .then(([res1, res2, res3]) => {
      setResults({ method1: res1.data, method2: res2.data, method3: res3.data })
    })
    .catch(err => console.error(err))
    .finally(() => setLoading(false))
}, [])
```

## 응답 타입 지정

axios 제네릭을 활용하여 응답 타입을 명시합니다.

```tsx
// ✅ 타입 지정된 API 호출
const res = await api.get<DashboardStats>('/dashboard/stats')
const stats: DashboardStats = res.data

// ❌ 타입 미지정
const res = await api.get('/dashboard/stats')
const stats = res.data  // 타입 추론 불가
```

## 캐싱 전략

현재 프로젝트에는 캐싱 레이어가 없습니다.
TanStack Query 또는 SWR 도입을 권장하지만, 도입 전까지는
`useEffect` 의존성 배열을 정확히 관리하여 불필요한 재요청을 방지합니다.

## 핵심 규칙 요약

| 규칙 | 내용 |
|------|------|
| HTTP 클라이언트 | `api` (from `@/lib/api`) 만 사용 |
| 패턴 | `useEffect` + `useState` |
| 병렬 요청 | `Promise.all` |
| 에러 처리 | `try/catch` + `console.error` (UI 에러 상태 추가 권장) |
| 응답 타입 | axios 제네릭으로 명시 |
| URL 하드코딩 | 금지 — `NEXT_PUBLIC_API_URL` 은 `api.ts` 에서만 사용 |
